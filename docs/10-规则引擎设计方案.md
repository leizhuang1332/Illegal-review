# 规则引擎设计方案

## 一、设计目标与定位

### 1.1 核心目标
规则引擎作为视频违规审核系统的核心决策组件，负责处理具有明确模式的违规内容，提供确定性判断，支持灵活的规则配置和动态扩展。

### 1.2 定位与边界
- **上游依赖**：图像识别引擎、音频分析引擎、文本分析引擎
- **下游输出**：融合决策层
- **职责边界**：专注于规则匹配和确定性判定，不涉及概率推理

### 1.3 设计原则
- **可扩展性**：支持规则动态添加、修改、删除，无需重启服务
- **可配置性**：规则通过配置文件或API进行管理
- **高性能**：支持快速规则匹配，满足高吞吐场景
- **可审计性**：完整记录规则执行过程，便于追溯

### 2.2 模块划分

| 模块名称 | 职责描述 | 核心能力 |
|----------|----------|----------|
| **规则解析模块** | 解析规则配置，构建规则执行树 | 语法解析、规则编译、表达式求值 |
| **规则管理模块** | 规则的增删改查、版本管理 | 规则生命周期管理、版本控制、导入导出 |

| **规则执行模块** | 根据匹配结果执行相应动作 | 动作执行、结果聚合、冲突处理 |
| **规则存储层** | 持久化规则配置 | 规则存储、版本管理、缓存机制 |

## 三、技术选型

### 3.1 核心技术方案

| 模块 | 技术方案 | 备选方案 | 选型理由 |
|------|----------|----------|----------|
| **规则语言** | 自定义DSL + JSON Schema | Drools、CLIPS | 自定义DSL更灵活，易于集成 |
| **规则存储** | PostgreSQL + Redis | MongoDB | PostgreSQL支持复杂查询，Redis用于缓存 |
| **表达式引擎** | Python表达式求值 | PyExpr、SymPy | 轻量级，易于扩展 |
| **规则匹配** | Rete算法变种 | 线性匹配 | 支持复杂规则的高效匹配 |
| **版本管理** | 数据库版本记录 | Git版本控制 | 便于追溯和回滚 |

### 3.2 规则类型支持

| 规则类型 | 描述 | 匹配策略 |
|----------|------|----------|
| **黑名单规则** | 明确禁止的内容模式 | 精确匹配、模糊匹配、正则匹配 |
| **白名单规则** | 明确允许的内容模式 | 精确匹配、前缀匹配、后缀匹配 |
| **阈值规则** | 基于数值阈值的判定 | 比较运算、范围判断 |
| **组合规则** | 多条件逻辑组合 | AND/OR/NOT逻辑组合 |

## 四、各模块详细设计

### 4.1 规则解析模块

**模块职责**：将规则配置解析为可执行的规则对象

#### 输入参数

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| rule_config | dict | 规则配置字典 | 符合规则JSON Schema |
| rule_id | string | 规则唯一标识 | 非空 |
| version | string | 规则版本号 | 语义化版本 |

#### 输出参数

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| parsed_rule | RuleObject | 解析后的规则对象 | 包含条件和动作 |
| condition | Expression | 条件表达式 | 可执行的表达式对象 |
| action | Action | 动作定义 | 包含动作类型和参数 |
| metadata | dict | 规则元数据 | 包含规则ID、版本、描述 |

#### 解析流程
1. **Schema验证**：验证规则配置符合JSON Schema
2. **条件解析**：将条件字符串解析为抽象语法树
3. **动作解析**：解析动作类型和参数
4. **规则编译**：生成可执行的规则对象

---

### 4.2 规则管理模块

**模块职责**：管理规则的全生命周期

#### 输入参数（规则创建/更新）

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| name | string | 规则名称 | 非空，最大100字符 |
| type | string | 规则类型 | "blacklist" / "whitelist" / "threshold" / "composite" |
| description | string | 规则描述 | 可选，最大500字符 |
| condition | string | 条件表达式 | 符合DSL语法 |
| action | dict | 动作配置 | 包含type和params |
| priority | int | 规则优先级 | 0-100，数字越大优先级越高 |
| enabled | bool | 是否启用 | 默认true |
| tags | list | 标签列表 | 用于分类和搜索 |

#### 输出参数（规则查询）

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| rules | list | 规则列表 | 每个元素为规则详情 |
| rule_id | string | 规则ID | 唯一标识 |
| name | string | 规则名称 | - |
| type | string | 规则类型 | - |
| status | string | 规则状态 | "enabled" / "disabled" |
| priority | int | 规则优先级 | - |
| created_at | datetime | 创建时间 | - |
| updated_at | datetime | 更新时间 | - |

#### 核心功能
- **规则创建**：验证配置并保存到存储层
- **规则更新**：更新规则配置，创建新版本
- **规则删除**：逻辑删除，保留历史版本
- **规则启用/禁用**：动态控制规则生效状态
- **规则导入导出**：支持批量导入导出规则配置

---

### 4.3 规则匹配引擎

**模块职责**：执行规则匹配逻辑，确定是否命中规则

#### 输入参数

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| video_features | dict | 视频特征数据 | 包含图像、音频、文本特征 |
| rules | list | 规则列表 | 待匹配的规则对象 |
| video_id | string | 视频标识 | 非空 |

#### 输出参数

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| matched_rules | list | 命中的规则列表 | 每个元素包含rule_id和匹配详情 |
| match_info | dict | 匹配详情 | 包含匹配路径、匹配值 |
| match_type | string | 匹配类型 | "exact" / "partial" / "range" |
| confidence | float | 匹配置信度 | 对于模糊匹配，范围[0,1] |

#### 匹配策略

| 规则类型 | 匹配策略 | 示例 |
|----------|----------|------|
| **黑名单** | 精确匹配 | 敏感词精确命中 |
| **黑名单** | 模糊匹配 | 敏感词变体匹配 |
| **黑名单** | 正则匹配 | 正则表达式匹配 |
| **白名单** | 精确匹配 | 认证账号ID匹配 |
| **白名单** | 前缀匹配 | 域名前缀匹配 |
| **阈值** | 比较匹配 | 数值大于阈值 |
| **阈值** | 范围匹配 | 数值在范围内 |
| **组合** | 逻辑匹配 | 多条件AND/OR组合 |

---

### 4.4 规则执行模块

**模块职责**：根据匹配结果执行相应动作

#### 输入参数

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| matched_rules | list | 命中的规则列表 | 来自匹配引擎 |
| video_features | dict | 视频特征数据 | 原始输入特征 |
| video_id | string | 视频标识 | 非空 |

#### 输出参数

| 参数名 | 类型 | 描述 | 约束 |
|--------|------|------|------|
| decision | string | 决策结果 | "pass" / "reject" / "pending" |
| rule_id | string | 命中的规则ID | 白名单/黑名单命中时 |
| action_type | string | 执行的动作类型 | "allow" / "block" / "review" |
| evidence | dict | 决策证据 | 包含匹配的特征和规则详情 |
| processing_time | float | 处理耗时（毫秒） | >0 |

#### 动作类型

| 动作类型 | 描述 | 适用场景 |
|----------|------|----------|
| **allow** | 允许通过 | 白名单命中 |
| **block** | 拒绝通过 | 黑名单命中、阈值超标 |
| **review** | 人工复核 | 组合规则命中、需要进一步审查 |
| **log** | 记录日志 | 仅记录不影响决策 |

#### 冲突处理策略
- **优先级策略**：优先级高的规则优先执行
- **类型优先策略**：黑名单 > 白名单 > 阈值 > 组合规则
- **最新规则优先**：同优先级时，最新更新的规则优先

---

### 4.5 规则存储层

**模块职责**：持久化规则配置和版本管理

#### 存储结构

| 表/集合 | 用途 | 关键字段 |
|----------|------|----------|
| **rules** | 规则主表 | rule_id, name, type, condition, action, priority, status |
| **rule_versions** | 规则版本历史 | rule_id, version, config, created_at |
| **rule_execution_logs** | 规则执行日志 | rule_id, video_id, result, execution_time |

#### 缓存策略
- **规则缓存**：将启用的规则缓存到Redis，减少数据库查询
- **缓存更新**：规则变更时自动更新缓存
- **缓存失效**：设置合理的过期时间，定期刷新

## 五、规则定义规范

### 5.1 规则配置JSON Schema

```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["name", "type", "condition", "action"],
  "properties": {
    "name": {"type": "string", "maxLength": 100},
    "type": {"enum": ["blacklist", "whitelist", "threshold", "composite"]},
    "description": {"type": "string", "maxLength": 500},
    "condition": {"type": "string"},
    "action": {
      "type": "object",
      "required": ["type"],
      "properties": {
        "type": {"enum": ["allow", "block", "review", "log"]},
        "params": {"type": "object"}
      }
    },
    "priority": {"type": "integer", "minimum": 0, "maximum": 100},
    "enabled": {"type": "boolean", "default": true},
    "tags": {"type": "array", "items": {"type": "string"}}
  }
}
```

### 5.2 条件表达式DSL

**支持的操作符**：

| 操作符类型 | 操作符 | 描述 | 示例 |
|------------|--------|------|------|
| **比较** | `>`, `<`, `>=`, `<=`, `==`, `!=` | 数值比较 | `visual.exposure_ratio > 0.3` |
| **逻辑** | `AND`, `OR`, `NOT` | 逻辑运算 | `A AND B OR NOT C` |
| **字符串** | `CONTAINS`, `STARTS_WITH`, `ENDS_WITH`, `MATCHES` | 字符串操作 | `text.content CONTAINS "敏感词"` |
| **集合** | `IN`, `NOT IN` | 集合包含判断 | `face.identity IN ["person_a", "person_b"]` |
| **正则** | `REGEX_MATCH` | 正则匹配 | `text.content REGEX_MATCH "^.*敏感.*$"` |

**特征路径规范**：

| 特征类型 | 路径前缀 | 示例 |
|----------|----------|------|
| **图像特征** | `visual.` | `visual.exposure_ratio`, `visual.object_count` |
| **音频特征** | `audio.` | `audio.sensitive_score`, `audio.event_count` |
| **文本特征** | `text.` | `text.sensitive_word_count`, `text.content` |
| **人脸特征** | `face.` | `face.count`, `face.identity` |

### 5.3 规则配置示例

**黑名单规则**：
```json
{
  "name": "敏感人物识别",
  "type": "blacklist",
  "description": "检测视频中是否包含敏感人物",
  "condition": "face.identity IN [\"politically_sensitive_person_1\", \"politically_sensitive_person_2\"]",
  "action": {"type": "block", "params": {"reason": "检测到敏感人物"}},
  "priority": 100,
  "enabled": true,
  "tags": ["政治", "人脸"]
}
```

**阈值规则**：
```json
{
  "name": "画面暴露度超标",
  "type": "threshold",
  "description": "画面暴露比例超过30%判定为违规",
  "condition": "visual.exposure_ratio > 0.3",
  "action": {"type": "block", "params": {"reason": "画面暴露度超标"}},
  "priority": 80,
  "enabled": true,
  "tags": ["色情", "图像"]
}
```

**组合规则**：
```json
{
  "name": "疑似色情内容",
  "type": "composite",
  "description": "画面暴露+性暗示动作+敏感字幕同时满足",
  "condition": "visual.exposure_ratio > 0.2 AND visual.sexual_posture_score > 0.7 AND text.sensitive_word_count > 0",
  "action": {"type": "review", "params": {"reason": "疑似色情内容，需人工复核"}},
  "priority": 90,
  "enabled": true,
  "tags": ["色情", "组合"]
}
```

## 十、关键设计要点

### 10.1 性能优化策略

| 优化方向 | 策略 | 预期收益 |
|----------|------|----------|
| **规则缓存** | 将启用规则缓存到Redis | 减少数据库查询，提升匹配速度 |
| **规则优先级** | 按优先级排序，快速路径优先 | 减少不必要的规则匹配 |
| **索引优化** | 为规则条件字段建立索引 | 加速规则查询 |
| **批量匹配** | 支持批量视频特征匹配 | 提升吞吐量 |

### 10.2 容错与降级机制

- **规则解析失败**：跳过错误规则，记录日志
- **存储层异常**：使用缓存中的规则继续运行
- **规则冲突**：按优先级和类型策略处理
- **超时控制**：设置规则匹配超时时间

### 10.3 可观测性

- **规则执行日志**：记录每条规则的执行结果
- **性能监控**：监控规则匹配耗时、命中率
- **规则覆盖率**：统计规则被触发的频率
- **告警机制**：规则配置错误、匹配异常时告警

## 十二、总结

规则引擎采用**模块化、可扩展**架构，通过规则解析→规则匹配→规则执行的流程，实现对视频特征的确定性判定。核心特点：

- **可扩展性**：支持规则类型、操作符、数据源的动态扩展
- **高性能**：规则缓存、优先级排序、批量匹配优化
- **可配置性**：支持通过API动态管理规则，无需重启服务
- **可审计性**：完整的规则执行日志和版本管理

该设计满足视频违规审核场景对规则引擎的核心需求，既能快速处理明确违规模式，又能灵活适应业务规则的变化。
